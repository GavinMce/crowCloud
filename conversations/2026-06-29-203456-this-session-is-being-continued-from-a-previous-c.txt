● Pushed. Here's the full release flow:

  ---
  How releases work

  1. Every push to main — release-plz.yml runs:
  - If there are unreleased conventional commits (feat:, fix:, etc.), it
  creates/updates a "Release crowCloud v0.x.x" PR with the version bump in
  Cargo.toml and an updated CHANGELOG.md
  - When you merge that PR, release-plz creates the git tag v0.x.x

  2. Tag push triggers publish.yml — four parallel jobs:
  - Docker images → ghcr.io/gavinmce/crow-api:0.1.0, :0.1, :latest (same for
  operator, vps-agent). Uses cargo-chef in the Dockerfiles for proper layer
  caching.
  - CLI binaries — crow cross-compiled for:
    - x86_64-unknown-linux-gnu (native)
    - aarch64-unknown-linux-gnu (via cross)
    - x86_64-apple-darwin (macOS 13 runner)
    - aarch64-apple-darwin (macOS 14 / M1 runner)
  - Helm chart → oci://ghcr.io/gavinmce/charts/crowcloud:0.1.0
  - GitHub Release — created after the above finish, with CLI tarballs attached
  and auto-generated release notes

  One setup step required: Create a GitHub fine-grained PAT with Contents: write
  scope and store it as RELEASE_PLZ_TOKEN in repo secrets. Without it the
  workflow falls back to GITHUB_TOKEN, but tags created by GITHUB_TOKEN won't
  trigger publish.yml (GitHub security restriction). With the PAT it all chains
  automatically.

● Branch is up. Here's a summary of what was built:

  ---
  crow-provider-proxmox — 5 modules, ~350 lines of real implementation:

  client.rs — ProxmoxClient wrapping reqwest
  - PVEAPIToken auth header on every request
  - Typed get / post / post_multipart / put / delete with automatic {"data": 
  ...} envelope unwrapping
  - wait_task(upid, timeout_secs) — decodes the node from the UPID, polls
  /tasks/{upid}/status every 2s until status == "stopped", checks exitstatus == 
  "OK"

  vm.rs — the main event
  - create_vm: nextid → full clone → snippet uploads → config
  (cores/memory/net/cicustom/ipconfig) → disk resize → start, all task-awaited
  - Cloud-init: uploads vm-{vmid}-user.yaml and vm-{vmid}-network.yaml as
  snippets via multipart; falls back to ipconfig0=ip=<ip>/24,gw=<.1> if only a
  bare IP is provided; DHCP if nothing
  - delete_vm: best-effort stop + purge with destroy-unreferenced-disks=1

  volume.rs — honest about the Proxmox constraint: no standalone volume API
  exists, so create_volume records intent as storage:name:sizeG in the handle
  for drivers to use at VM creation time. delete_volume deletes a real volid
  when given one.
